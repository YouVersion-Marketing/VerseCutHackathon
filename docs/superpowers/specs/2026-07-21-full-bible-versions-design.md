# Full Bible version catalog — design

## Goal

Expand the studio's language/version picker from today's ~70 curated languages
(each with a single default version) to the **full YouVersion catalog**: every
language tag in the provided CSV is selectable, and each language exposes **all**
of its versions. Verse text keeps flowing through the license-safe internal
reader API.

Source of truth: `versions 3(in).csv` — 3,823 rows of `(version_id,
language_tag)`, with no names. Names, English language labels, and text
direction are enriched from the internal `version.json` endpoint.

## Decisions (confirmed with user)

- **Scope:** all languages + all versions (widest reach).
- **Pipeline:** a static, committed manifest generated by a one-time build
  script. No per-selection network cost.
- **Language labels:** English name only (e.g. "Spanish").
- **Default version per language:** reuse today's curated default from
  `appLanguages.ts` where it exists, else the lowest version id.
- **Delivery:** the ~230 KB versions map ships as a JSON asset fetched once on
  load (not bundled into the JS).
- **Generation:** run the full fetch during implementation to commit the real
  manifest.

## Data pipeline

### Approach (chosen): CSV-authoritative + `version.json` enrichment

The CSV is the authoritative id set. A build script fetches internal
`version.json?id=X` for each id and reads `title`, `abbreviation`,
`local_title`, `local_abbreviation`, and the `language` object
(`name`, `iso_639_1`, `language_tag`, `text_direction`). The heavy `books`
array (~95 KB/response) is discarded.

Rejected alternatives:
- **Platform-export join** (alfred `export_language_bible_mapping.py`): the
  Platform key is license-scoped and omits/403s many languages, leaving
  hundreds of ids without names. Insufficient coverage.
- **`versions.json` catalog crawl**: verified to return only a *subset*
  (`language_tag=eng` returned 2 versions). Incomplete.

### Build script — `scripts/build-bible-manifest.ts`

- Reads a committed copy of the CSV at `scripts/data/bible-versions.csv`
  (checked in so regeneration is reproducible).
- Fetches `version.json` per id with a **concurrency cap (~20)** and simple
  retry; logs and **skips** ids that 404 or error, printing a summary count.
- Uses the same internal headers as `internalProvider` / alfred `client.py`.
- Groups results by `language_tag`; resolves the default version; sorts
  versions by abbreviation.
- Writes `public/bible-manifest.json`.
- Wired as an npm script `bible:manifest`; **not** part of `npm run build`
  (network-heavy, one-shot).

Pure, network-free helpers (CSV parse, grouping, default resolution, sort) live
in `src/lib/bible/manifestBuild.ts` so they run in the normal Vitest suite; the
script layers fetch/IO on top.

## Manifest shape

`public/bible-manifest.json`:

```jsonc
{
  "languages": [
    { "tag": "eng", "code": "en", "name": "English", "dir": "ltr",
      "defaultVersionId": "1", "versionCount": 42 }
  ],
  "versionsByTag": {
    "eng": [
      { "id": "1",  "abbr": "KJV",   "title": "King James Version" },
      { "id": "12", "abbr": "ASV",   "title": "American Standard Version" }
    ]
  }
}
```

- `tag` — `language_tag` (e.g. iso_639_3-ish); the **stable, unique** key.
- `code` — `iso_639_1` when the endpoint provides one, else falls back to
  `tag`. Drives downstream locale features (voices, CTA, fonts).
- `name` — English `language.name`.
- `dir` — `text_direction` (`ltr`/`rtl`), for RTL rendering.
- `defaultVersionId` — curated default (see below) else lowest id.
- `versionCount` — number of versions for the tag (lets the hybrid picker label
  a group without loading `versionsByTag`).
- `versions` carry `{ id, abbr, title }` only (localized fields dropped since
  labels are English-only; `title` is the English title).

### Default-version resolution

1. Build a map `curatedVersionId → language_tag` by resolving each
   `appLanguages.ts` default id's `language_tag` from its fetched `version.json`.
2. For each tag group, if a curated default id belongs to that tag, use it;
   otherwise use `min(id)`.

`appLanguages.ts` is **retained** as the curated-default seed consumed by the
build script only.

## Provider & UI integration

The picker is data-driven and already async, so changes are mostly in the data
layer.

### `internalProvider.ts` (manifest-backed)

- On first use, fetch `/bible-manifest.json` once via a **memoized promise**
  (module-level cache), same pattern as today's `versionCache`.
- `listLanguages()` → `manifest.languages` mapped to
  `{ id: tag, name, code, dir }` (English name).
- `listVersions(tag)` → `manifest.versionsByTag[tag]` mapped to
  `{ id, abbreviation: abbr, name: title, languageId: tag }`. **Synchronous
  from cached data** — faster than today, which network-fetched the single
  default.
- `listBooks` / `fetchPassage` unchanged (still hit chapter/version endpoints
  per selected version id).

### `types.ts`

Add optional `code?: string` to `Language` so the iso_639_1 code travels with
the picker entry instead of being parsed out of the (now tag-based) id. This
preserves distinctions like `zh_CN` vs `zh_TW` that pure iso_639_1 would
collapse.

### `hybridProvider.ts`

- Delegates to `internalProvider`, so it inherits the full manifest for free;
  passes `code` through on languages and keeps `i:`/`p:` id prefixing.
- **Recommendation (flag for review):** drop the Platform-catalog layer — the
  manifest now supersedes it and avoids Platform 403s. "Top picks" grouping is
  kept. If preferred, the Platform layer can remain untouched (`p:` ids are
  independent).

### `useStudio.ts`

`languageCode` currently parses the code out of the picker id. Change it to use
the **selected language's `code`** (falling back to the parsed id when absent),
so voices/CTA/fonts keep receiving iso_639_1 codes for tag-keyed languages.

## Guardrails (out of scope)

- No change to rendering, compositing, or verse-text fetching.
- No mobile-specific picker changes; the existing searchable dropdown handles
  the larger list.
- No localized (non-English) version titles in the UI.
- No live/automatic catalog refresh — regeneration is a manual `bible:manifest`
  run.

## Testing

Unit tests (fixture-based, no live network) in
`src/lib/bible/manifestBuild.test.ts`:
- CSV parse (ids + tags, header handling).
- Grouping by tag.
- Default resolution: curated-wins-else-lowest.
- Version sort by abbreviation.

`appLanguages.test.ts` and `internalProvider.test.ts` stay green. `npm run
check` green. Manually verify in-browser: language list is large, picking a
language lists all its versions, a verse fetches for a non-curated version, and
an RTL language renders correctly.
